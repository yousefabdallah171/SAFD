## 2. PROBLEM & PURPOSE

### Problem Statement - v1.0
Social media managers and small business owners spend 5+ hours daily manually responding to comments and engaging with followers across Facebook and Instagram. This manual process is time-consuming, inconsistent, and causes them to miss opportunities to convert engaged followers into customers during off-hours. No automation exists that uses official APIs only (not scraping), respecting Meta's compliance rules.

### Problem Statement - v2.0 (Expansion)
v1.0 users still spend 1-2 hours daily writing reply text, even with automation enabled. Managing multiple business accounts requires switching between platforms. No visibility into which automated interactions convert to sales. Mobile users can't manage accounts on-the-go. Agencies managing 10+ clients have no way to handle them in one platform.

### Project Purpose - v1.0
Build a multi-tenant SaaS platform that automates comment replies and DM follow-ups based on user-defined rules, allowing each business to manage their social media engagement at scale without manual intervention using only official Meta APIs.

### Project Purpose - v2.0
Expand v1.0 with AI-powered reply generation (removing manual writing), native mobile app (manage anytime/anywhere), multi-tenant account switching (manage multiple businesses seamlessly), CRM integration (track conversions), and advanced analytics (prove ROI).

### Business Value - v1.0
- **Time Savings:** Reduces manual engagement time by 80% (5+ hours/day → 1 hour/day)
- **Response Rate:** Increases response rate to 100% (24/7 automation)
- **Conversion:** Improves conversion from followers → customers via automated DM funnels
- **Scalability:** Allows users to manage unlimited posts and automate unlimited replies

### Business Value - v2.0
- **Additional Time Savings:** AI replies reduce remaining 1-2 hours to 10 minutes/day (90% reduction)
- **Mobile Engagement:** 24/7 mobile access increases engagement by 300%
- **Multi-Business Scale:** Agencies can manage 10+ clients in one platform
- **Conversion Tracking:** CRM integration shows exact ROI (comment → DM → lead → sale)
- **Monetization:** Unlock Premium tier ($49/month) with AI, advanced analytics, CRM

### Opportunity
v1.0 captures SMB market. v2.0 targets agencies (10x higher LTV) and enterprises. AI replies are top feature request. 40% of users want mobile access. CRM integration is key to proving ROI and increasing retention.

## 3. GOALS & OBJECTIVES

### Primary Goal - v1.0
Deliver a fully functional multi-tenant SaaS dashboard by end of Q2 2026 that allows users to:
1. Register and authenticate
2. Add their Meta API credentials securely
3. View all Facebook posts and Instagram posts/reels
4. Configure unlimited automation rules per post/reel
5. Automatically reply to comments and send DMs based on triggers
6. Track and analyze all engagement with real-time metrics

### Primary Goal - v2.0
Enhance v1.0 with:
1. AI-powered smart reply generation (user chooses API provider or uses SAFD default)
2. Native mobile app (iOS + Android) with core features
3. Multi-tenant account switching for managing multiple businesses
4. CRM integration (HubSpot, Pipedrive, Salesforce) to track conversions
5. Advanced analytics dashboard with ROI visualization
6. All within 99.9% uptime and < 2s load times

### Objectives - v1.0

2. **Integration:** Seamlessly connect to Facebook & Instagram via official Meta APIs with proper OAuth flow

4. **Analytics:** Real-time dashboard showing posts, engagement, reply rates, DM delivery, conversions
5. **Reliability:** 99.9% uptime with proper error handling, retry logic, and queue-based processing

## 4. SCOPE

### In Scope - v1.0

**Frontend:**
- Landing page (marketing, CTA, features overview)
- Registration & login page
- Email verification flow
- Dashboard with sidebar navigation
- API credential management page (secure input for user's Meta API keys)
- Onboarding wizard (5-step guide to add API and first post)
- Posts & reels listing page (paginated, searchable)
- Per-post automation rule builder (UI for keywords, reply text, DM message)
- Global rules configuration (apply rules to all posts)
- Activity/logs dashboard (real-time feed of comments, replies, DMs)
- Analytics dashboard (engagement metrics, response rates, conversions, charts)
- Settings page (account, integrations, billing placeholder)

**Backend:**
- User authentication (email/password registration, JWT login)
- User profile and account management
- Tenant isolation (each user's data completely separate)
- Meta OAuth integration (Facebook & Instagram login)
- API credential encryption and secure storage
- Fetch posts/reels from Meta Graph API
- Webhook receiver for incoming comments (Meta Webhooks)
- Rule matching engine (keyword-based triggers)
- Comment auto-reply service
- Auto-DM messaging service
- Activity logging and tracking
- Analytics aggregation (engagement, reply rates, DM delivery)
- Error handling and retry logic (exponential backoff)
- Rate limit handling (queue-based processing)
- Database and caching (PostgreSQL + Redis)

**Database:**
- Users table (multi-tenant base)
- Tenants table (account/workspace per user)
- Meta credentials table (encrypted API tokens)
- Posts table (synced from Meta)
- Reels table (synced from Meta)
- Automation rules table (per-post and global)
- Comments table (received via webhook)
- Replies table (auto-replies sent)
- DM messages table (auto-DMs sent)
- Activity logs table (tracking all actions)
- Analytics snapshots table (historical metrics)

**Infrastructure:**
- REST API with versioning (/api/v1/)
- Webhook endpoint for Meta callbacks
- Job queue (BullMQ + Redis) for processing
- Database migrations and seeding
- CI/CD pipeline (GitHub Actions)
- Docker containerization
- Deployment ready (VPS/AWS/Railway)

### Assumptions - v1.0
1. Users have Meta Business accounts with admin access
2. Users will provide valid Meta API credentials
3. Meta APIs remain stable and available
4. Users comply with Meta's platform policies
5. Database backups are handled by hosting provider
6. SSL/TLS certificates managed by hosting

### Assumptions - v2.0
1. Users will integrate with at least one CRM (HubSpot, Pipedrive, Salesforce)
2. Users have valid OpenAI/Claude/Gemini API keys or use SAFD's default with monthly limit
3. Mobile users have iOS 14+ or Android 10+ devices
4. Multi-tenant users manage up to 10 accounts
5. CRM sync happens within 5 minutes of contact creation
6. Push notifications work on 95%+ of devices
7. Users' AI API keys are valid and funded

### Constraints - v1.0
1. **Meta API Compliance:**
   - Must use official Meta Graph API only (no scraping)
   - Must respect 24-hour messaging window on Facebook
   - Must comply with Instagram comment reply rules
   - Must not trigger spam detection or account restrictions

2. **Rate Limits:**
   - Meta API has rate limits per app (typically 200 calls/user/hour)
   - Must implement queue and throttling to prevent hitting limits
   - Batch requests where possible

3. **Regulatory:**
   - Must comply with GDPR (if EU users)
   - Must encrypt stored credentials
   - Must not store or expose user API keys in logs

4. **Technical:**
   - Frontend must load dashboard in < 2 seconds
   - API responses must be < 500ms
   - Database queries optimized for 100K+ records per user
   - Webhook processing must be async and fast

### Constraints - v2.0
1. **AI API Costs:** Must track and enforce monthly limits per tenant
2. **Mobile Size:** App bundle < 50MB (Android), < 100MB (iOS)
3. **Battery Life:** Mobile app must use < 5% battery per hour of use
4. **Data Privacy:** All API keys encrypted, no logs contain credentials
5. **Mobile Performance:** First-load < 3 seconds, API responses < 500ms
6. **Compliance:** GDPR, CCPA for EU/US users, proper data retention policies
7. **CRM Rate Limits:** Respect CRM API rate limits (HubSpot 500/min, Salesforce 15K/24h)

### Dependencies - v1.0
1. Meta Graph API availability and uptime
2. Meta OAuth for account connection
3. PostgreSQL database
4. Redis for caching and queuing
5. Email service (SendGrid/Mailgun) for verification
6. Hosting platform (Vercel, Railway, AWS, or self-hosted VPS)

#### P0-F014: Error Handling & Retry Logic

**Description:** System gracefully handles API errors, network failures, rate limits, and expired tokens with exponential backoff retry strategy and clear error logging.

**User Story:**
As a user, I want the system to be resilient to failures so that temporary issues don't break automation and I'm notified clearly when action is needed.

**Trigger:** Any API call fails, network error occurs, or rate limit hit

**Pre-conditions:**
- Job in queue encounters error
- System has retry strategy configured

**Post-conditions:**
- Job retried with exponential backoff
- Error logged with timestamp and reason
- User notified if manual action needed

**Main Flow:**
1. Worker executes job (send_comment_reply, send_dm, fetch_posts, etc.)
2. API call fails with error:
Error: 429 Too Many Requests
3. System checks error type:
   - **429 (Rate Limit):** Retry with backoff
   - **401 (Unauthorized):** Log error, notify user, don't retry
   - **400 (Bad Request):** Log error, don't retry
   - **5xx (Server Error):** Retry with backoff
   - **Network Error:** Retry with backoff
4. If retriable error:
   - Calculate retry delay: 2^attempt_count (1s, 2s, 4s, 8s, 16s, 32s, 60s max)
   - Increment retry_count
   - If retry_count < 3: Re-queue job with delay
   - If retry_count >= 3: Mark as FAILED, log error, notify user
5. Log entry created:
   - INSERT INTO error_logs (job_id, error_type, error_message, retry_count, next_retry_at, tenant_id)
6. User notified via dashboard:
   - If 3+ retries failed: Banner "DM to @user_123 failed after 3 retries. Check connection."
7. Retry logic applied across all operations:
   - Fetching posts
   - Sending replies
   - Sending DMs
   - Authenticating with Meta

**Alternate Flows:**
- **AF1 - Partial Success:** If 1 out of 5 DMs fail, others succeed, only failed DM retried
- **AF2 - Circuit Breaker:** If 50%+ of jobs fail in last hour, halt new jobs, alert user
- **AF3 - Manual Retry:** User can manually retry failed job from dashboard

**Acceptance Criteria:**
- [ ] Exponential backoff implemented (1s, 2s, 4s, 8s, 16s, 32s, 60s)
- [ ] Max 3 retries before giving up
- [ ] Error type identified and appropriate action taken
- [ ] Retriable vs non-retriable errors distinguished
- [ ] Errors logged with full context (job_id, error, timestamp, tenant_id)
- [ ] User notified of permanent failures (401, etc.)
- [ ] User notified after max retries exceeded
- [ ] Retry history visible in logs
- [ ] Manual retry option available
- [ ] Dead letter queue for unprocessable jobs
- [ ] Alerts triggered for high error rates

#### P0-F015: Queue-Based Processing

**Description:** All async jobs (comment replies, DMs, syncs) queued in Redis using BullMQ for reliable, rate-limited processing without blocking API requests.

**User Story:**
As a user, I want the system to process comments and send replies reliably without hitting Meta's rate limits, so automation runs smoothly 24/7.

**Trigger:** Comment received, API rate limit approached, job queued

**Pre-conditions:**
- Redis running and accessible
- BullMQ installed and configured
- Queue workers running

**Post-conditions:**
- Job added to queue
- Worker picks up job and processes
- Job completed or retried

**Main Flow:**
1. Comment webhook received → system queues job immediately:
```javascript
   await commentQueue.add('process_comment', {
     comment_id: '123',
     post_id: '456',
     user_id: '789',
     message: 'Great product!'
   }, { priority: 1, attempts: 3, backoff: { type: 'exponential', delay: 1000 } });
```
2. Return 200 OK to Meta within 100ms (non-blocking)
3. Worker processes job:
   - Retrieve job from queue
   - Match comment against rules
   - If matched, queue 'send_comment_reply' and 'send_dm' jobs
4. Jobs can be queued with:
   - **Priority:** High priority jobs (critical replies) processed first
   - **Delay:** Scheduled for later execution
   - **Attempts:** Max retry count
   - **Backoff:** Exponential backoff on failure
5. Worker configuration:
   - Multiple workers (concurrency = 5) to handle 100+ comments/hour
   - Rate limiting: 10 API calls/second max (enforced by queue)
   - Memory efficient: Jobs removed from Redis after completion
6. Monitoring:
   - Queue depth visible in dashboard
   - Worker status (idle/busy) tracked
   - Job completion rate monitored
7. Job lifecycle:
   - `waiting` → `active` → `completed` / `failed`
   - Failed jobs: `failed` → `wait` (if retry configured) → `active` again

**Alternate Flows:**
- **AF1 - Queue Full:** If queue has 1000+ pending jobs, new webhook returns 429 (graceful degradation)
- **AF2 - Worker Crash:** Job in progress marked as failed, goes back to queue, another worker picks it up
- **AF3 - Redis Unavailable:** Queue operations fail, webhook returns 500, retry logic kicks in

**Acceptance Criteria:**
- [ ] Redis queue operational and monitored
- [ ] Jobs queued with priority, delay, backoff configured
- [ ] Worker concurrency: 5 (process 5 jobs simultaneously)
- [ ] Rate limiting: < 10 API calls/second
- [ ] Job completion logged
- [ ] Failed jobs retried with exponential backoff
- [ ] Queue depth visible in admin panel
- [ ] Worker logs available for debugging
- [ ] Dead letter queue for unprocessable jobs
- [ ] Job TTL: 7 days (jobs expire if not processed)
- [ ] Performance: Process 100+ comments/hour without congestion

#### P0-F026: Tenant Isolation Verification

**Description:** Backend verification that all queries are properly filtered by tenant_id, preventing any cross-tenant data access.

**User Story:**
As a user, I want complete assurance that my data is completely isolated from other users so that I can trust my sensitive information is protected.

**Trigger:** Every API request, database query execution

**Pre-conditions:**
- User authenticated with JWT containing tenant_id
- Database has all tables with tenant_id column
- Row-level security (RLS) configured

**Post-conditions:**
- Query validated against JWT tenant_id
- Only user's tenant_id data returned
- 403 Forbidden if tenant_id mismatch

**Main Flow:**
1. User makes API request: `GET /api/v1/posts`
2. Backend receives JWT with:
```json
   {
     "sub": "user_123",
     "tenant_id": "tenant_456",
     "iat": 1234567890
   }
```
3. Middleware validates tenant_id in JWT
4. Database query executed:
```sql
   SELECT * FROM posts 
   WHERE tenant_id = :tenant_id 
   AND user_id = :user_id
   LIMIT 100
```
5. Only posts from tenant_456 returned
6. If user tries to manually query different tenant:
GET /api/v1/posts?tenant_id=tenant_789
7. System detects mismatch:
   - JWT has tenant_id = tenant_456
   - Query requests tenant_id = tenant_789
   - Returns 403 Forbidden: "Unauthorized access to resource"

8. Verification runs on ALL operations:
   - SELECT queries: WHERE tenant_id filter applied
   - INSERT: tenant_id set from JWT
   - UPDATE: WHERE tenant_id filter applied
   - DELETE: WHERE tenant_id filter applied

9. Database indexes created:
```sql
   CREATE INDEX idx_posts_tenant_id ON posts(tenant_id);
   CREATE INDEX idx_posts_tenant_user ON posts(tenant_id, user_id);
   CREATE INDEX idx_comments_tenant ON comments(tenant_id);
```

10. Audit logging:
INSERT INTO audit_logs (user_id, tenant_id, action, resource, result, timestamp)
VALUES ('user_123', 'tenant_456', 'GET', 'posts', 'SUCCESS', NOW())

**Alternate Flows:**
- **AF1 - Token Tampering:** If JWT modified, signature validation fails, 401 Unauthorized
- **AF2 - Missing Tenant ID:** If JWT missing tenant_id, 403 Forbidden
- **AF3 - Admin Check:** If user tries admin routes, 403 Forbidden (even with valid JWT)

**Acceptance Criteria:**
- [ ] All database tables have tenant_id column
- [ ] All SELECT queries include WHERE tenant_id filter
- [ ] All INSERT queries set tenant_id from JWT
- [ ] All UPDATE/DELETE queries include WHERE tenant_id filter
- [ ] Middleware validates tenant_id on every request
- [ ] Cross-tenant access returns 403 Forbidden
- [ ] Audit logs record all access attempts
- [ ] Database indexes on (tenant_id, resource_id) exist
- [ ] Performance: Indexed queries < 100ms even with 1M+ records
- [ ] Tests verify isolation with multiple test tenants
- [ ] No tenant can query other tenant's data via API
- [ ] No tenant can modify other tenant's data
